# Transcript: Multi-Turn Conversation with Query Rewriting (Phase 3)

---

## 1. Requirements

**User:**
> "Read this documentation, right now is just a basic RAG implementation. The user query is passed on and the top few chunks retrieved and an llm will answer back their questions. Now i want the ability to continue the conversation. What are some techniques available to continue the conversation?"

**Context:**
Phase 1 (CLI with chunk viewer) and Phase 2 (LLM integration) were complete. The chatbot was fully stateless — each query had no memory of prior turns. The user wanted to enable multi-turn conversations where follow-up questions are understood in context.

---

## 2. Design Overview

Multi-turn conversation requires solving two problems:

1. **LLM Context Problem**: Claude needs to remember prior Q&A exchanges
2. **Retrieval Problem**: Vague follow-ups like "tell me more about that" need better vector search

The solution combines two techniques:

```
Turn 1:
  User Query → Retrieve(query) → Generate(query, chunks, []) → Store in history

Turn 2+:
  User Query → Rewrite(query, history) → Retrieve(rewritten_query) → Generate(query, chunks, history) → Store in history
```

**Key Insight**: Query rewriting fixes the RAG retrieval gap. Without it, "tell me more about the europe guide" produces bad vector search results because the embedding model has no context.

---

## 3. Design Decisions

### Technique Selection: Query Rewriting + History

**Alternatives Considered:**

1. **Naive History Only** — Pass all prior turns to LLM
   - ✓ Simple to implement
   - ✗ Doesn't fix retrieval problem (vague queries still return poor chunks)

2. **Query Rewriting + History** ✓ (SELECTED)
   - ✓ Fixes both retrieval AND LLM context problems
   - ✓ Better UX for follow-up questions
   - ✓ Rewritten query used for retrieval, original stored in history (natural phrasing)
   - ✗ Adds ~0.5–1s latency on turn 2+ (extra LLM call)

3. **Sliding Window History** — Keep only last N turns
   - ✓ Limits token growth
   - Can be combined with #2 for very long sessions

4. **Summary Memory** — Compress old turns
   - ✓ Best for 100+ turn sessions
   - ✗ Complex to implement

**Decision Rationale**: Query rewriting directly addresses the RAG problem. For typical interactive sessions, the extra 0.5–1s latency is acceptable given the improved retrieval quality.

### History Storage Strategy

**Decision**: Store conversation history as a list of `{role, content}` dicts, only plain Q&A (no chunk context)

**Justification:**
- Keeps messages list compact as turns accumulate
- Chunk context injected only in current turn (not stored in history)
- History grows by exactly 2 entries per turn (predictable)
- Clean format aligns with Claude Messages API

```python
history = [
    {"role": "user", "content": "What is the Budget Travel Guide to Europe?"},
    {"role": "assistant", "content": "The Budget Travel Guide to Europe covers..."},
    {"role": "user", "content": "tell me more about the europe one"},
    {"role": "assistant", "content": "The Budget Travel Guide features..."},
]
```

### Query Rewriting Timing

**Decision**: Only rewrite on turn 2+; turn 1 skips rewriting entirely

**Justification:**
- Turn 1 has no history, so no rewriting needed
- Eliminates unnecessary latency on first query
- Rewriting only helps when there's context to incorporate
